# Data Models

This document describes the key data structures and file formats used within the PBA Vision Mapping system. Most data is persisted in CSV or NumPy-specific file formats.

## 1. Configuration Data

*   **Source**: `src\recipes\active_recipe.csv` (and potentially other recipe files).
*   **Format**: CSV.
*   **Accessed by**: Various components via `read_recipe_csv()` methods (e.g., `ImageAcquisitionModule`, `GlassCertificateGenerationModule`, `ErrorMapPlottingModule`).
*   **Description**: Contains key-value pairs for system parameters, thresholds, file paths, and other operational settings. The exact schema can vary based on the active recipe but generally includes parameters like pixel-to-mm conversion ratios, default paths, etc.

## 2. Log Files

*   **Example Filename**: `Log_file_2D_expansion_X_axis.csv` (pattern: `Log_file_<scan_type_details>.csv`), `Log_file_1D.csv`, `Log_file_auto_focus.csv`.
*   **Generated by**: `ImageAcquisitionModule` (`scan_and_capture.py`) via `create_log_file()` and methods like `log_write()`, `expantion_log_write()`, `log_write_in_position_test()`.
*   **Format**: CSV.
*   **Header (as defined in `create_log_file`)**: `Time,Run,image name,Y position,X position,Y comand position,X comand position,Z comand position,axis_x_position_error,axis_y_position_error,axis_z_position_error, pix_Y,pix_X, pix_R, temp0, temp1, temp2,temp3`
*   **Description**: Records detailed information during scanning operations. Columns include:
    *   `Time`: Timestamp of the log entry (e.g., `YYYY-MM-DD-HH-MM-SS`).
    *   `Run`: The current iteration number of the scan.
    *   `image name`: Filename of the captured image (e.g., `0.png`, `1.png`).
    *   `Y position`: Target index for Y-axis (row index `A`).
    *   `X position`: Target index for X-axis (column index `B`).
    *   `Y comand position`: Commanded Y coordinate from `location_array` or `expantion_location_array`.
    *   `X comand position`: Commanded X coordinate from `location_array` or `expantion_location_array`.
    *   `Z comand position`: Commanded Z coordinate from `location_array` or `expantion_location_array`.
    *   `axis_x_position_error`: Position error of X-axis read from motor controller (`get_PE()`).
    *   `axis_y_position_error`: Position error of Y-axis read from motor controller (`get_PE()`).
    *   `axis_z_position_error`: Position error of Z-axis read from motor controller (`get_PE()`).
    *   `pix_Y`: Y pixel coordinate of the detected dot center from `self.calculated_values[0]` (result of `pe_calculation.calculate_center()` on the captured image).
    *   `pix_X`: X pixel coordinate of the detected dot center from `self.calculated_values[1]`.
    *   `pix_R`: Radius of the detected dot from `self.calculated_values[2]`.
    *   `temp0`, `temp1`, `temp2`, `temp3`: Temperature readings from `read_temparature.read_temparature()`.

## 3. Calibration and Reference Data

### 3.1. Calculated Reference Locations
*   **Filename**: `calculated_all_referance_locations.csv`
*   **Generated by**: Likely an external process or initial setup, used as input by `GlassCertificateGenerationModule`.
*   **Format**: CSV.
*   **Description**: Stores the nominal/ideal X and Y coordinates of dots on the calibration glass/jig.
    *   Typically, each row represents a dot, with columns for X_position, Y_position.

### 3.2. Glass Certificate
*   **Filename**: `glass_certificate_2D_relative.csv` (or similar, e.g., `glass_certificate.csv`)
*   **Generated by**: `GlassCertificateGenerationModule` (`generate_2D_glass_certificate_62207.py`).
*   **Format**: CSV.
*   **Description**: Contains the differences between nominal dot positions (from `calculated_all_referance_locations.csv`) and the machine's centered encoder positions for those dots. The 'measured' values are derived from the 'Y comand position' and 'X comand position' columns in the vision log file (e.g., `Log_file_2D_expansion_X_axis.csv`), which are recorded after the self-centering routine has updated the `expantion_location_array` and commanded a move to these corrected coordinates. Columns typically include:
    *   `Nominal_X`, `Nominal_Y`: Ideal coordinates from `calculated_all_referance_locations.csv`.
    *   `Measured_X`, `Measured_Y`: Encoder X, Y coordinates when the machine believes it is centered on the dot, taken from the vision log's commanded positions.
    *   `Delta_X`, `Delta_Y`: Difference (`Measured - Nominal`).
*   **Note**: While camera calibration data (`camera_calibration_data.npz`) is an input to the `generate_2D_glass_certificate_62207.py` script, its application for correcting `Measured_X, Measured_Y` for optical distortions is identified as a placeholder in the script. The current `Measured_X, Measured_Y` reflect encoder readings corresponding to vision-based centered positions without further optical correction.

### 3.3. Dot Center Encoder Locations (Expansion Array)
*   **Example Filename**: `Expansion_array_2D_0.csv`, `Expansion_array_positive{run}.csv`, `Expansion_array_negative{run}.csv`.
*   **Generated by**: `ImageAcquisitionModule` (`scan_and_capture.py`) during centering scan methods (e.g., `scan_method_4`, `scan_method_5`, `scan_method_8`), saved via `save_expansion_array()`.
*   **Format**: CSV. The NumPy array `self.expantion_location_array` (shape: 3, rows, columns) is reshaped to 2D (`-1`, `self.expantion_location_array.shape[-1]`) before saving. This means each row in the CSV corresponds to a dot, and columns are likely X, Y, Z corrected coordinates.
    *   Example: If `expantion_location_array` stores X, Y, Z for each dot, the CSV will have rows like: `corrected_X_dot1, corrected_Y_dot1, corrected_Z_dot1`
*   **Description**: Stores the final corrected X, Y, and Z encoder coordinates where the system centered on each dot during an expansion/centering scan. This represents the machine's precise understanding of each dot's location after visual correction.
    *   The array `self.expantion_location_array` is updated by `estimate_dot_center_button_clicked()` with `correction_value_x` and `correction_value_y` for the X and Y planes, and by `auto_focus()` for the Z plane.

### 3.4. Camera Calibration Data
*   **Filename**: `camera_calibration_data.npz`
*   **Generated by**: An external camera calibration process.
*   **Format**: NumPy NPZ file.
*   **Accessed by**: `GlassCertificateGenerationModule`.
*   **Description**: Stores camera intrinsic and extrinsic parameters, distortion coefficients, etc., necessary for correcting image distortions and accurate metrology.

## 4. Error Map Data

The `EncoderErrorMatrixModule` (`generate_2D_encoder_error_matrix.py`) produces several files during the generation of encoder error compensation maps. These include intermediate calculations and final output files.

### 4.1. Intermediate Cumulative Distance Matrices
*   **Filenames**:
    *   `vision_x_cumulative_distance_matrix.csv`
    *   `vision_y_cumulative_distance_matrix.csv`
    *   `encoder_x_cumulative_distance_matrix.csv`
    *   `encoder_y_cumulative_distance_matrix.csv`
*   **Generated by**: `EncoderErrorMatrixModule` (`generate_2D_encoder_error_matrix.py`).
*   **Format**: CSV.
*   **Description**:
    *   `vision_x_cumulative_distance_matrix.csv` & `vision_y_cumulative_distance_matrix.csv`: Store cumulative distances between grid points along X and Y axes, respectively. These distances are derived from the `glass_certificate_2D_relative.csv` (specifically, the `Delta_X` and `Delta_Y` columns, which represent the calibrated vision system's measurement of dot spacing relative to nominal). They represent an ideal grid as measured and corrected by the vision system.
    *   `encoder_x_cumulative_distance_matrix.csv` & `encoder_y_cumulative_distance_matrix.csv`: Store cumulative distances between grid points along X and Y axes, respectively. These distances are derived from the `Expansion_array_2D_0.csv` (or equivalent file specified as `dot_center_encoder_location_matrix_file_path`). They represent the actual encoder movements recorded when visiting each dot.

### 4.2. Raw Encoder Error Maps (Pre-Camera Rotation Correction)
*   **Filenames**:
    *   `x_original_camera_angel_acs_format_error_map_2D.csv`
    *   `y_original_camera_angel_acs_format_error_map_2D.csv`
*   **Generated by**: `EncoderErrorMatrixModule`.
*   **Format**: CSV.
*   **Description**: Contain the initial calculated X and Y error values at each grid point. These errors are the difference between the encoder's cumulative travel and the vision system's cumulative travel *before* any correction for camera mounting angle is applied. Each cell corresponds to a grid point and holds the error value. These are primarily for diagnostic purposes or as an intermediate step.

### 4.3. Corrected 2D Encoder Error Maps (Post-Camera Rotation Correction)
*   **Filenames**:
    *   `x_converted_camera_angel_acs_format_error_map_2D.csv`
    *   `y_converted_camera_angel_acs_format_error_map_2D.csv`
*   **Generated by**: `EncoderErrorMatrixModule`.
*   **Format**: CSV.
*   **Description**: These files store the final calculated error values for each axis across the 2D grid, *after* applying corrections for camera mounting angle (derived from `get_camera_rotation_matrix` and applied via `convert_original_camera_angel_to_converted_camera_angel_error_map`). The structure is a 2D array where each cell corresponds to a grid point and holds the error value. These are the primary human-readable error maps representing the machine's positioning error.

### 4.4. ACS Formatted Error Maps
*   **Filenames**:
    *   `XERRO2D.DAT`
    *   `YERRO2D.DAT`
*   **Generated by**: `EncoderErrorMatrixModule` (via `generate_ACS_errormaps()` method).
*   **Format**: ACS controller specific format (text-based, typically space-separated values, with a specific header and structure defined by ACS requirements).
*   **Description**: Data prepared for direct loading into the ACS motion controller's 2D error compensation buffers. These files are generated from the corrected 2D error maps (`x_converted_camera_angel_acs_format_error_map_2D.csv` and `y_converted_camera_angel_acs_format_error_map_2D.csv`) and formatted according to the ACS controller's specifications.

## 5. Image Data

*   **Source**: `src\images\` (or other configured image storage locations).
*   **Format**: PNG, TIFF, or other image formats as configured.
*   **Accessed by**: Primarily `ImageAcquisitionModule`, but potentially other modules needing to access captured imagery.
*   **Description**: Contains the actual image data captured during the inspection or mapping process. The images are used for both real-time analysis and archival purposes. Image filenames and organization are typically defined by the active recipe and the specifics of the scanning procedure.
