# Requirements: PBA Vision Mapping

This document consolidates all features, user stories, non-functional requirements, and clarifying questions for the PBA Vision Mapping project.

---

## 1. Features

### FEAT-001: Dot Grid Scanning for Glass Scale Calibration
**Description:** Enables the system to perform a scan of the glass scale using `scan_method_5()`. This involves the user selecting the "2D expansion" test procedure, inputting parameters such as `First target`, `Last target`, `Number of runs`, `Number of Images`, and `wait time` via the "Testing" tab in the UI, and initiating the scan. The output is a "Log_file" containing raw scan data.

**Acceptance Criteria:**
- `AC1`: User can select the "2D expansion" test procedure from the "Select test" dropdown in the "Testing" tab.
- `AC2`: User can input `First target`, `Last target`, `Number of runs`, `Number of Images`, and `wait time` parameters.
- `AC3`: System successfully executes `scan_method_5()` (from `scan_and_capture.py`) upon clicking the "Scan" button.
- `AC4`: A "Log_file" is generated containing the data from the scan.
- `AC5`: UI shows scan progress.

---

### FEAT-002: Glass Scale Certificate Generation
**Description:** Enables the system to process the "Log_file" (generated by `FEAT-001`) to produce a glass scale calibration certificate. This is done via the "Error mapping" tab by selecting the log file and running the `generate_2D_glass_certificate_62207.py` script.

**Acceptance Criteria:**
- `AC1`: User can select the "Log_file" in the "Import file" section of the "Error mapping" tab.
- `AC2`: The `generate_2D_glass_certificate_62207.py` script successfully processes the selected "Log_file" upon clicking "Run Actions".
- `AC3`: A `glass_certificate.csv` file is generated by the system.
- `AC4`: The generated certificate enables dot location determination with a precision meeting the target of 100nm for dot distance.

---

### FEAT-003: Encoder Error Matrix Generation
**Description:** Allows the system to use the calibrated `glass_certificate.csv` (from `FEAT-002`) to identify and quantify errors originating from the encoder. This is done via the "Error mapping" tab by selecting the certificate and running the `generate_2D_encoder_error_matrix.py` script. It processes vision-based data to generate a 2D error matrix for both X and Y axes, outputting them as CSV files.

**Acceptance Criteria:**
- `AC1`: User can select the `glass_certificate.csv` in the "Import file" section of the "Error mapping" tab.
- `AC2`: The `generate_2D_encoder_error_matrix.py` script successfully uses the glass scale certificate as an input upon clicking "Run Actions".
- `AC3`: The script processes vision-based image data to accurately calculate encoder errors.
- `AC4`: The script generates an `x_error_map_2D.csv` file containing the calculated X-axis error data.
- `AC5`: The script generates a `y_error_map_2D.csv` file containing the calculated Y-axis error data.
- `AC6`: UI shows progress and paths to the generated CSV files.

---

### FEAT-004: Error Correction System Testing, Validation & Controller File Generation
**Description:** Provides a robust method to test and validate the error correction capabilities of the system. This involves using `generate_2D_error_mapping_test.py` to process `x_error_map_2D.csv` and `y_error_map_2D.csv`. The script can remove specified rows/columns to produce `x_error_map_2D_for_test.csv` and `y_error_map_2D_for_test.csv` (controller correction files). It also allows for one of these generated test CSV files to be designated as the "uploaded" map to simulate controller input, and then uses the remaining data for validation of the correction. This is initiated from the "Error mapping" tab.

**Acceptance Criteria:**
- `AC1`: User can initiate the process from the "Error mapping" tab using the "Run Correction Test & Validation" button.
- `AC2`: The script can successfully process `x_error_map_2D.csv` and `y_error_map_2D.csv`.
- `AC3`: The script can remove specified rows and/or columns from the input maps to produce `x_error_map_2D_for_test.csv`.
- `AC4`: The script can remove specified rows and/or columns from the input maps to produce `y_error_map_2D_for_test.csv`.
- `AC5`: The system allows for one of these generated test CSV files to be designated as the "uploaded" map to the controller via UI input.
- `AC6`: The system can use the remaining (non-uploaded) part of the original error map to validate the effectiveness of the applied corrections.
- `AC7`: UI shows progress and validation results.

---

### FEAT-005: Error Map Visualization
**Description:** Allows users (e.g., Engineers) to visualize the measured X-directional and Y-directional error maps through graphical plots. This aids in understanding error patterns, verifying data integrity, and assessing the outcome of the correction process.

**Acceptance Criteria:**
- `AC1`: The `generate_2D_error_mapping_test_plot.py` script can generate a visual plot for X-directional errors.
- `AC2`: The script can generate a visual plot for Y-directional errors.
- `AC3`: The generated plots are clear, legible, and accurately represent the error data.

---

### FEAT-006: Y-Directional Data Refactoring for Enhanced Plotting
**Description:** Addresses the current challenge in plotting Y-directional repeatability data. This feature involves refactoring how this specific data is saved or processed to make it directly and easily usable by visualization tools.

**Acceptance Criteria:**
- `AC1`: The data format for Y-directional errors in the relevant CSV file is modified, or an intermediate processing step is implemented to transform it.
- `AC2`: The refactored Y-directional data can be directly consumed by the `generate_2D_error_mapping_test_plot.py` script.
- `AC3`: The Y-directional plot generated using the refactored data accurately and clearly reflects the repeatability characteristics.

---

### FEAT-007: Integrated UI for Core System Operations
**Description:** Integrates the core Python processing scripts (scanning, certificate generation, error matrix generation, testing & validation, visualization) into the main PySide6 graphical user interface. This will allow users to trigger these operations and view their progress/results directly from the UI, rather than running scripts manually.

**Acceptance Criteria:**
- `AC1`: User can initiate Dot Grid Scanning (`FEAT-001`) from the UI.
- `AC2`: User can initiate Glass Scale Certificate Generation (`FEAT-002`) from the UI.
- `AC3`: User can initiate Encoder Error Matrix Generation (`FEAT-003`) from the UI.
- `AC4`: User can initiate Error Correction System Testing, Validation & Controller File Generation (`FEAT-004`) from the UI.
- `AC5`: User can initiate Error Map Visualization (`FEAT-005`) from the UI.
- `AC6`: UI provides feedback on the status and completion of these operations.

## 2. User Stories

### US-001
**Story:** "As an Engineer, I want to perform a dot grid scan using the UI so that I can collect the necessary raw data for glass scale calibration."
- `AC1`: Given I am on the "Testing" tab, when I select the "2D expansion" test and input parameters, then I can click "Scan" to initiate `scan_method_5()`.
- `AC2`: Given the scan is initiated, when the process completes, then a "Log_file" containing scan data is generated.

### US-002
**Story:** "As an Engineer, I want to generate an accurate glass scale certificate from a scan log file so that I have a precise and reliable reference for measuring and correcting encoder errors."
- `AC1`: Given I have a "Log_file" from a previous scan, when I select it and initiate generation, then a `glass_certificate.csv` is created by `generate_2D_glass_certificate_62207.py`.
- `AC2`: Given the certificate is generated, it confirms that dot locations can be determined with an accuracy of at least 100nm.

### US-003
**Story:** "As an Engineer, I want to generate a 2D encoder error matrix using the glass scale certificate so that I can accurately quantify the geometric inaccuracies of the encoder system."
- `AC1`: Given a valid `glass_certificate.csv`, when I run the error matrix generation, then `x_error_map_2D.csv` is produced.
- `AC2`: Given a valid `glass_certificate.csv`, when I run the error matrix generation, then `y_error_map_2D.csv` is produced.

### US-004
**Story:** "As an Engineer, I want to test the error correction system by generating controller-specific correction files, simulating their application, and validating the results."
- `AC1`: Given full error maps, when I initiate the test process, then controller-ready CSV files are created.
- `AC2`: Given a test CSV, when a portion is designated for controller upload, then the remaining portion is used for validation.
- `AC3`: Given corrections are applied, then the system provides a clear comparison between expected and actual corrected values.

### US-005
**Story:** "As an Engineer, I want to generate clear visual plots of both X and Y directional error maps so that I can easily inspect error patterns and assess correction effectiveness."
- `AC1`: Given an X-axis error map CSV, when I run the plotting script, then an X-directional error plot is generated.
- `AC2`: Given a Y-axis error map CSV (refactored), when I run the plotting script, then a Y-directional error plot is generated.

### US-006
**Story:** "As a Developer, I want to refactor the Y-directional error data handling so that it can be seamlessly used by plotting tools."
- `AC1`: The Y-error data is stored in a format directly compatible with the plotting script.
- `AC2`: The plotting script runs successfully without requiring complex data manipulation steps.

### US-007
**Story:** "As an Engineer, I want an integrated user interface to perform all core operations so that I can manage the entire error mapping workflow efficiently from a single application."
- `AC1`–`AC5`: I can initiate all core operations (scanning, certificate, error matrix, testing, visualization) from the UI.
- `AC6`: The UI provides clear feedback on the status and completion of these operations.

## 3. Non-Functional Requirements (NFRs)

| NFR ID | Category | Requirement |
|:---|:---|:---|
| `NFR-ACCU-001` | Accuracy | Glass scale certificate generation must achieve dot location accuracy of 100 nanometers. |
| `NFR-USAB-001` | Usability | Generated error map plots must be clear, correctly scaled, and accurately represent error data. |
| `NFR-MAINT-001` | Maintainability | All Python scripts in the processing pipeline should be modular, well-commented, and follow consistent coding standards. |
| `NFR-RELI-001` | Reliability | Error map generation must be deterministic and repeatable — identical inputs must produce identical outputs. |
| `NFR-PERF-001` | Performance | Data processing scripts should execute within a reasonable timeframe. Specific time targets TBD based on dataset sizes. |
| `NFR-USAB-002` | Documentation | The complete process flow must be clearly documented with diagrams and explanations of each step. |
| `NFR-COMPAT-001` | Compatibility | Generated CSV files for the ACS controller must be compatible with the ACS controller's expected input format. |
